# pm-prd-writing

产品经理需求文档技能。输入产品想法和用户痛点，通过专业需求挖掘方法，生成完整PRD文档和任务清单。

## 使用场景

- 从0到1的新产品PRD撰写
- 功能迭代需求文档编写
- 用户故事梳理与优先级排序
- 需求评审材料准备

## 输入

用户提供：
- **产品名称/背景**：产品是什么，解决什么问题
- **目标用户**：谁会用这个产品
- **核心功能点**：想要实现的主要功能（可粗糙）
- **用户痛点**：用户目前遇到的问题
- **交付格式**：飞书云文档 + 飞书多维表格

## 核心方法论

### 三问法
```
1. 用户遇到了什么问题？（问题描述）
2. 用户现在怎么解决的？（现有方案）
3. 你建议怎么解决？（解决方案）
```

### JTBD (Jobs-to-be-Done)
```
用户"雇用"产品来完成的工作：
- 功能性工作（我要完成什么任务）
- 情感性工作（我想有什么感觉）
- 社会性工作（我想别人怎么看我）
```

### 用户行为四大驱动力
```
🔵 推力（PUSH）：用户当前的不满/痛点
🟢 拉力（PULL）：用户向往的理想状态
🟡 焦虑（ANXIETY）：用户对新方案的担忧
🔴 习惯（HABIT）：用户现有的固定行为模式
```

### 用户故事格式
```
作为 [角色]，
我想 [做某件事]，
以便 [实现某个目标]。

验收标准：
- Given [前置条件]
- When [操作]
- Then [预期结果]
```

## 工作流程

### Step 1: 需求挖掘
```
使用三问法 + JTBD 框架：
1. 明确用户是谁
2. 挖掘用户真正想完成的工作
3. 识别推力/拉力/焦虑/习惯
4. 区分真需求和伪需求
```

### Step 2: 用户故事梳理
```
将需求转化为用户故事：
1. 角色识别（谁在用）
2. 场景还原（什么情况下）
3. 目标明确（想完成什么）
4. 验收标准（怎么算成功）
```

### Step 3: 功能拆解
```
按 MECE 原则拆解功能：
- 核心功能（必须有的）
- 重要功能（应该有的）
- 扩展功能（可以有的）
- 未来功能（以后再做的）
```

### Step 4: PRD文档生成
```
生成完整PRD，包含：
# 产品需求文档 (PRD)

## 1. 概述
## 2. 用户与场景
## 3. 用户故事地图
## 4. 功能需求详情
## 5. 非功能性需求
## 6. 优先级与迭代计划
## 7. 附录
```

### Step 5: 任务清单生成
```
生成飞书多维表格格式的任务清单：
| 任务名称 | 所属故事 | 优先级 | 负责人 | 状态 |
|---------|---------|--------|-------|------|
```

### Step 6: 交付飞书
```
1. 调用 lark-doc 创建PRD云文档
2. 调用 lark-base 创建任务清单多维表格
3. 返回文档和表格链接
```

## 依赖工具

- `lark-doc` - 创建飞书云文档写入PRD
- `lark-base` - 创建飞书多维表格写入任务清单
- `browser` - 搜索竞品PRD参考

## 注意事项

1. **需求真伪**：区分用户说的和用户真正需要的
2. **优先级依据**：优先级排序需有明确理由
3. **验收标准**：每条用户故事需有可测试的验收标准
4. **边界情况**：不要遗漏异常流程
5. **可追溯**：用户故事之间需有逻辑关联

## 验证方式

- PRD包含7大核心章节
- 用户故事数量 ≥ 3个
- 每个故事有明确的验收标准
- 功能有优先级排序
- 成功写入飞书云文档 + 多维表格
